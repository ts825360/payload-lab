# Loupe 프로젝트의 프롬프트 설계 참고 자료

## 문서 목적

해커톤 팀원 Romain Hartmann의 [loupe](https://github.com/RomainHartmann/loupe) 저장소에서 사용된 프롬프트/에이전트 지시서를 조사하고, 우리 캡스톤 프로젝트(attack-lens)에 응용할 수 있는 패턴을 정리한 문서입니다. 원문은 전부 영어이며, 이 문서는 구조와 원칙을 한국어로 요약·정리한 것입니다 (원문 그대로의 복사가 아님).

Claude와의 대화에서 조사한 내용을 기반으로 작성했습니다.

---

## 1. 앱 내부 LLM 프롬프트 — `analysis-prompt.ts`, `translation-prompt.ts`

loupe는 기사 신뢰성 검증 엔진으로 5개 AI 제공자(Grok, Claude, Gemini, ChatGPT, Mistral)에 동일한 프롬프트를 보내고 합의된 결과만 사용합니다. 이때 프롬프트 두 개(텍스트 분석용, 번역 대조용)에서 반복되는 설계 원칙이 있습니다.

### 1-1. "Hard rules" 섹션으로 금지사항을 명시적으로 못박기

프롬프트 중간에 "Hard rules"라는 별도 섹션을 두고, 모델이 하기 쉬운 실수를 미리 금지합니다.

- 원문을 절대 다시 쓰거나 고치지 말 것 (요약·교정·의역 금지)
- 자기 확신도를 숫자(%)로 반환하지 말 것 — 가짜 정밀도를 만들지 않기 위함
- "이 주장은 참/거짓이다"라고 판정하지 말고, "왜 검토가 필요한지"만 설명할 것
- 해당 사항이 없으면 억지로 결과를 만들지 말고 빈 값을 반환할 것

**왜 중요한가**: 이런 규칙이 없으면 LLM은 "도움이 되려는 성향" 때문에 원문을 손대거나, 근거 없이 확신도를 지어내거나, 애매한 경우에도 억지로 판정을 내리는 실패 패턴을 보입니다. 이걸 프롬프트에서 원천 차단합니다.

### 1-2. 출력 형식을 스키마로 완전히 고정

자연어 응답이 아니라 `{"findings": [...]}` 형태의 JSON만 반환하도록 강제하고, 각 항목의 필드명(`type`, `quote`, `target`, `evidence`)과 `type`에 들어갈 수 있는 카테고리 값까지 열거형(enum)으로 프롬프트 안에 명시합니다.

**왜 중요한가**: 파싱 실패를 원천 차단하고, 프론트엔드가 항상 같은 모양의 데이터를 받는다고 신뢰할 수 있게 합니다.

### 1-3. "검증 가능한 근거"를 항상 요구

`quote` 필드는 항상 원문에서 그대로 가져온 문자열(verbatim)이어야 한다고 못박습니다. 이렇게 하면 나중에 코드로 "이 quote가 실제로 원문에 존재하는가"를 기계적으로 검증할 수 있어, LLM이 존재하지 않는 문장을 지어내는 환각(hallucination)을 잡아낼 수 있습니다.

### 우리 프로젝트에 적용할 점

우리 Lens는 AI가 아니라 규칙 기반이지만, 같은 원칙을 코드 설계에 적용할 수 있습니다.

- Lens가 반환하는 진단 메시지는 항상 "어떤 조건을, payload의 어느 부분에서 확인했는지" 근거를 함께 반환하게 만들 것 (막연한 텍스트 설명 금지)
- Lens 결과 포맷을 처음부터 고정된 스키마로 정의할 것 (이미 진행 중인 [모듈 경계 이슈](https://github.com/ts825360/attack-lens/issues/3), [Lens 조건 체인 이슈](https://github.com/ts825360/attack-lens/issues/9)와 연결됨)
- "해당 사항 없음이면 억지로 만들지 말 것" 원칙은 우리 프로젝트의 [Lens 폴백 이슈](https://github.com/ts825360/attack-lens/issues/6)와 정확히 반대 방향이므로 주의: loupe는 "판단 불가능하면 침묵"이 맞는 선택이지만, 우리는 "판단 불가능해도 체크리스트형 폴백은 반드시 보여준다"가 맞는 선택입니다 (침묵하면 학습자가 아무것도 못 얻기 때문). 같은 원칙이라도 제품 목적에 따라 반대로 응용해야 하는 사례입니다.

## 2. 에이전트 운영 지시서 — `AGENTS.md` / `CLAUDE.md`

로메인 팀은 Claude Code와 Codex가 같은 저장소에서 동시에 작업했습니다. 이때 두 에이전트가 같은 규칙을 따르도록 `AGENTS.md` 한 파일에 모든 규칙을 몰아넣고, `CLAUDE.md`는 그 파일을 그대로 참조(`@AGENTS.md`)한 뒤 Claude 전용 항목만 추가하는 구조를 썼습니다. 우리도 Codex와 Claude를 함께 쓰므로 그대로 응용 가능한 구조입니다.

### 2-1. 프로젝트 개요를 맨 위에 고정

이름, 목적, 팀 구성, 마감 기한, 우선순위(핵심 기능 > 디자인 > 나머지), 기술 스택을 지시서 맨 위에 박아둬서, 에이전트가 매번 맥락을 다시 추론하지 않게 합니다.

### 2-2. "무결성 신호(canary)" 트릭

모든 응답을 특정 기호로 시작하게 강제하고, 그 기호가 안 나오면 "지시서가 컨텍스트 압축 등으로 유실됐다"는 신호로 해석합니다. 대화가 길어질수록 초반 지시가 요약되며 사라질 위험이 있는데, 이를 감지하는 저비용 장치입니다.

### 2-3. 제품 불변 조건(Product invariants)을 별도 섹션으로 명시

"이 도구는 절대 사용자의 입력 텍스트를 수정하지 않는다", "이 도구는 진실을 판정하는 재판관이 아니라 보조자다" 같은, 절대 타협할 수 없는 제품 원칙을 별도 섹션으로 분리해뒀습니다. 기능 구현 방식은 바뀌어도 이 원칙은 안 바뀐다는 뜻입니다.

**우리 프로젝트라면**: "Lens는 성공 이유를 설명하지 않는다", "시각화는 실패를 진단하지 않는다", "이 앱은 로컬 Docker 밖으로 나가지 않는다" 같은 문장이 여기에 해당합니다.

### 2-4. 이슈/라벨 규칙을 기계적으로 못박기

우선순위 이모지(🔴🟠🟡🟢)와 라벨(P0~P3)이 반드시 1:1 매칭되어야 한다는 규칙, 모든 이슈에 type/area 라벨이 필수라는 규칙이 명시되어 있습니다. **이 부분은 이미 우리 attack-lens 저장소 라벨 체계에 그대로 반영했습니다.**

### 2-5. "묻지 말고 이슈부터 만들어라" 자동화 규칙

작업이 30분을 넘거나 여러 파일에 걸치거나 대화가 끝나도 남아야 하는 내용이면, 묻지 않고 바로 이슈를 만들라고 되어 있습니다. 반대로 15분 미만의 단순 질문이나 오타 수정에는 이슈를 만들지 말라는 기준도 함께 명시되어 있습니다.

### 2-6. Git worktree 강제

"대화 1개 = worktree 1개 = 브랜치 1개, main에는 초기 세팅 외에 직접 커밋 금지" 규칙으로, 여러 실험적 시도가 서로 꼬이는 것을 방지합니다.

### 2-7. 언어 규칙

코드/문서는 영어만 사용하고, 이슈와 PR 설명만 영어 원문 + 한국어 번역을 구분선 아래 병기하도록 되어 있습니다 (팀원 중 한 명이 영어를 못 읽었기 때문). 대화는 상대방의 언어로 답하되, 저장소에 쓰는 글은 항상 영어가 우선입니다.

### 2-8. 명확한 경계선(Boundaries)

`.env` 계열 비밀 파일은 어떤 방법으로도 절대 읽지 말라고 명시되어 있고, 새 값을 추가할 땐 `.env.example`에 템플릿만 추가하고 실제 값은 사용자가 직접 넣게 하라고 되어 있습니다.

### 우리 프로젝트에 적용할 점

- `AGENTS.md`(또는 `CLAUDE.md`) 파일을 만들어 프로젝트 개요, 제품 불변 조건, 이슈/라벨 규칙, Command Injection 랩처럼 위험한 코드를 다룰 때의 경계선을 명시해두면 좋습니다.
- 특히 "제품 불변 조건" 섹션은 Lens와 시각화의 역할이 헷갈리지 않도록 문서화하는 데 유용합니다.

## 3. 서브에이전트 파이프라인 — `.claude/agents/*.md`

가장 눈여겨볼 부분입니다. 로메인 팀은 4개의 서브에이전트를 2개의 파이프라인으로 묶어 썼습니다.

### 3-1. 디자인 파이프라인: `apple-design-lead` → `design-critic`

- **1단계 (`apple-design-lead`)**: 화면/컴포넌트가 필요할 때 먼저 실행. 여백, 반경, 타이포그래피, 모션까지 수치로 못박은 디자인 스펙을 만들어서 "구현할 때 추측이 필요 없게" 함. 모든 상태(빈 화면, 로딩, 에러, 성공, 비활성화, hover, focus)를 빠짐없이 다루라고 명시.
- **2단계 (`design-critic`)**: 구현이 끝난 뒤 실행. 1단계가 만든 스펙과 구현 결과를 대조하며 결함을 찾음. "칭찬 금지, 결함과 승인/반려 판정만 보고하라"고 명시되어 있어 자기검열 없이 엄격하게 검수하도록 역할이 분리되어 있음.

### 3-2. 품질 파이프라인: `crash-hunter` → `code-hardener`

- **1단계 (`crash-hunter`)**: 버그를 **찾기만** 하고 절대 고치지 않음. 코드를 실제로 실행해서 재현되는 버그와, 코드를 읽어서 증명 가능한 잠재적 버그(널 체크 누락, 경계값, 인코딩 문제 등)를 모두 찾아 목록으로 보고.
- **2단계 (`code-hardener`)**: `crash-hunter`의 목록을 받아서 각 항목의 근본 원인을 고치고, 그 실패 조건을 강제로 재현하는 테스트를 정확히 1개씩 추가. 목록에 없는 건 손대지 않음(범위 확산 방지).

**핵심 설계 원칙**: 코드를 짠 사람(또는 에이전트)이 그 코드를 스스로 검수하지 않게 역할을 완전히 분리합니다. 이렇게 하면 "내가 짠 코드니까 내가 보기엔 문제없어 보인다"는 자기검증의 함정을 피할 수 있습니다.

### 우리 프로젝트에 적용할 점

이 패턴이 우리 캡스톤 프로젝트에 가장 직접적으로 응용 가능합니다.

- **`lab-implementer` → `safety-reviewer`**: 새 공격 랩(특히 [Command Injection](https://github.com/ts825360/attack-lens/issues/4))을 구현하는 에이전트와, 그 결과물이 안전 설계 요구사항(허용 명령어 목록, 격리 범위, 호스트 접근 차단)을 지켰는지 적대적으로 검수하는 별도 에이전트를 분리하면 좋습니다.
- **`lens-rule-author` → `lens-rule-critic`**: Lens 조건 체인을 짜는 에이전트와, 그 조건 체인이 [회귀 테스트 fixture](https://github.com/ts825360/attack-lens/issues/10)를 다 통과하는지, 예상 밖 payload에도 폴백이 제대로 동작하는지 검수하는 에이전트를 분리하면, "규칙을 짠 사람이 그 규칙의 허점을 못 보는" 문제를 줄일 수 있습니다.

## 요약: 우리 프로젝트가 가져갈 수 있는 것

| 로메인의 패턴 | 우리 프로젝트 적용 |
| :-- | :-- |
| 프롬프트에 "Hard rules"로 금지사항 명시 | Lens 진단 메시지는 항상 근거(조건+payload 위치)를 동반하게 설계 |
| 출력을 JSON 스키마로 고정 | Lens/시각화 결과 포맷을 코드 작성 전에 스키마로 확정 (이미 이슈 진행 중) |
| `AGENTS.md`에 제품 불변 조건 명시 | "Lens는 성공을 설명 안 함" 등 원칙을 문서화 |
| 이슈 라벨 체계(우선순위 이모지=P0~P3, type/area 필수) | 이미 attack-lens 저장소에 반영 완료 |
| `.env` 등 비밀 파일 절대 열람 금지 명시 | 우리 `AGENTS.md` 작성 시 그대로 채용 권장 |
| crash-hunter(찾기만) / code-hardener(고치기만) 역할 분리 | lab-implementer / safety-reviewer 역할 분리 검토 |
| apple-design-lead(제안) / design-critic(적대적 검수) | lens-rule-author / lens-rule-critic 역할 분리 검토 |

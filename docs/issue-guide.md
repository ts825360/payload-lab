# 이슈 작성·운영 가이드 (초안 v0)

> 이 저장소의 **모든 작업은 이슈에서 시작하고 이슈로 끝난다.** 기획·결정·검증·구현·버그·문서·
> 시행착오 — 무엇이든 이슈로 남긴다. 이 문서는 "언제·어떻게 이슈를 쓰고 굴리는가"의 규칙이며,
> **사람과 AI 에이전트 둘 다** 이 규칙대로 이슈를 만들고 실행한다.
>
> 참고: 우리 저장소(#1–#26) + [RomainHartmann/loupe](https://github.com/RomainHartmann/loupe)의
> 이슈 체계에서 더 나은 방식을 합쳤다.
>
> ⚠️ **초안입니다.** 검토 후 확정/영어 번역하세요.

## 0. 왜 이슈 중심으로 일하나
- 의사결정과 **그 근거**가 한곳에 남아 추적 가능하다.
- 새 팀원·AI가 대화 없이 **본문만 읽고 착수**할 수 있다.
- "종이 위에서 설계했다가 실제 데이터에 안 맞았던" 과거 실패를 막기 위해, **검증을 결정보다 먼저** 이슈로 박는다.

# A부 · 작업 흐름 (기획 → 착수 → 검증 → 마무리)

## A1. 기획안에서 이슈까지
- **기획안 완성**: 사용자가 초기 기획안(거친 아이디어)을 던지면, **관련 스킬(A6 표)을 적극 활용해
  기획안을 완성**한다 — 빠진 범위·제약·엣지케이스를 채우고 구조화한다.
- **등록은 사용자에게 묻는다**: 기획안이 충분히 완성됐다고 판단되면, **"이슈로 등록할까요?"를
  사용자에게 질문**하고 승인받은 뒤 이슈로 등록한다. **AI가 임의로 등록하지 않는다.**
- **등록 후 계획 제안**: 등록한 뒤에는 그 기획을 실제로 진행할 **이후 계획(다음 이슈·순서)** 을 제안한다.
- **이슈에 없는 내용이 나오면**: 대화·작업 중 이슈에 없는 새 항목이 나오면 **이슈화 여부를 사용자에게
  묻고**, 추가·제안을 **적극적으로** 한다. 기록되지 않은 결정·작업이 흘러가게 두지 않는다.

## A2. 작업 착수 = 스파이크 먼저 (리스크 제거)
- **리스크·불확실성이 있는 이슈**는 착수 시 **스파이크를 먼저 구현**해 리스크(불확실한 설계·인터페이스·
  안전성)를 먼저 제거한다. (자명한 변경은 스파이크 없이 바로 진행 — 규모에 맞게.)
- 스파이크에는 관련 스킬을 쓴다: UI·다이어그램은 **코드 전에 목업(visualize/show_widget)** 으로,
  로직·안전성은 **prototype·tdd·diagnosing-bugs** 로 먼저 대본다 (A6 표).
- **스파이크가 실패하면** 혼자 밀어붙이지 말고, **사용자에게 대안을 제안**해 문제를 함께 해결한다
  (옵션 A/B 제시 → 결정 요청).
- 이는 핵심 규율 **"검증 후 결정"** 과 짝을 이룬다: 규칙·스키마·인터페이스는 **현실적·적대적 입력에
  대보기(스파이크) 전에는 신뢰하지 않는다.** `needs-spike` 이슈를 관련 `needs-decision`보다 **앞에** 둔다.
  - 예: 우리 #4(안전 설계)·#5(스키마)·#9(Lens 체인), loupe #22(end-to-end 스파이크).

## A3. 검증 규칙 (기능별 → 최종 통합)
- **모든 검증은 여러 개의 이슈로 나눠** 기능별로 구현한다. 기능 하나를 검증하며 그때그때 **문제점·버그를 수정**한다.
- **"모든 기능을 한 번에 검증"하는 통합 검증 단계는 프로젝트의 제일 마지막**에 둔다 — 앞에서 기능별로 이미
  걸러진 상태에서 전체를 최종 확인한다.
- 검증 이슈 라벨: 결정 전 검증은 `needs-spike`, 마지막 통합 검증은 `type: chore`(+ 높은 우선순위).
- 규모에 맞게: 작은 변경은 PR의 자동 테스트로 갈음하고, **별도 검증 이슈는 리스크·범위가 있을 때** 만든다.

## A4. 팀 협업 규칙 (브랜치 · assignees · 닫기)
- **팀 프로젝트에서는 팀원 이름으로 브랜치를 만들고 그 브랜치에서만** 작업한다 (형식 `<이름>/<기능>`, 예: `<이름>/path-traversal`).
- 작업을 **시작할 때 assignees**로 자신을 지정해 "내가 이 이슈를 잡았다"는 상태를 나타낸다.
- 작업이 **완료되면 이슈를 닫는다** (PR이면 `Closes #번호`로 머지 시 자동 종료).

## A5. 시행착오 기록 (댓글 → 절대 안 닫는 회고 이슈)
- **모든 시행착오는 해당 이슈의 댓글**로 그때그때 남긴다 — 무엇을 시도했고, 왜 틀렸고, 어떻게 고쳤는지.
- 프로젝트 **마지막에** 흩어진 시행착오 이야기와 **해결방안 논의 결과를 하나의 회고 이슈**로 모은다.
- 이 **회고 이슈는 절대 닫지 않는다** — 살아있는 기록으로 계속 열어둔다.

## A6. 단계·이슈 유형별 활용 스킬

이슈를 잡으면 유형·area에 맞는 스킬을 **적극 호출**한다. AI가 이슈를 실행할 때 이 표가 "언제 무슨 스킬을 쓸지"의 지침이 된다. (⭐ = 이번 프로젝트에서 실제로 사용해 검증된 조합)

| 이슈 유형 / area | 활용 스킬 | 언제 |
|---|---|---|
| 프론트·시각화 구현/리디자인 (`area: frontend`/`visualization`) | ⭐ `frontend-design` · `taste-skill` + `visualize`(show_widget) | 일반 UI·다이어그램은 `frontend-design`, **랜딩·포트폴리오·리디자인**은 `taste-skill`(브리프 먼저 읽는 안티-템플릿). 코드 전에 목업으로 방향 검증 |
| 애니메이션·모션 (`area: frontend`) | `animate` | hover·페이지 전환·모달·리스트 정렬 등 **모션**을 넣을 때 (이징·접근성 포함) |
| 검증·리뷰 이슈 (기능별·최종 통합) | ⭐ `code-review` (표준·스펙 2축) | A3 검증. 큰 변경/PR을 닫기 전 감사 |
| 취약 실행부·배포 안전성 (`area: infra`, 랩 추가) | `security-review` | 실행 위험이 있는 이슈(새 랩·배포)를 닫기 전 안전 확인 |
| 자료조사 이슈 (#14–17 류) | ⭐ `research` + WebSearch | 문안·근거를 1차 출처로 채울 때 |
| 스파이크·프로토타입 (`needs-spike`) | `prototype` · `tdd` · `diagnosing-bugs` | A2에서 로직·안전성을 실제로 대볼 때 |

> **규칙**: 없는 스킬을 지어내지 말고 **실제 설치·검증된 스킬만** 적는다. 새 스킬을 도입하면 이 표에 한 줄 추가한다. 어떤 스킬을 쓸지 애매하면 A2처럼 사용자에게 물어 결정한다.
> `taste-skill`·`animate`는 새로 설치했으나 아직 프로젝트에서 써보지 않았다(그래서 ⭐ 없음) — 처음 쓸 때 결과를 이슈 댓글로 남긴다(A5).

# B부 · 이슈를 어떻게 쓰나 (형식)

## B1. 언제 이슈를 만드나 — 하나의 이슈 = 하나의 관심사

| 상황 | type 라벨 | 제목 동사 예 |
|---|---|---|
| 새 기능/랩 구현 | `type: feature` | 구현한다 / Build / Implement |
| 계약·타입·인터페이스 정의 | `type: feature` | 정의한다 / Define |
| 착수 전 결정이 필요한 제안 | `type: idea` + `needs-decision` | 결정한다 / Decide |
| 결정 전 실제로 대봐야 하는 검증 | `type: idea`/`feature` + `needs-spike` | 검증한다 / 교차검증 / Verify |
| 버그 | `type: bug` | 고친다 / Fix |
| 문서 | `type: docs` | 문서화한다 / Document |
| 도구·설정·유지보수 | `type: chore` | 정리한다 / Set up |

원칙: **한 이슈는 한 가지 이유로만 바뀐다.** 관심사가 여럿 섞이면 나눈다.

## B2. 제목 규칙 — 이슈탭에서 한눈에 띄게
형식: **`[우선순위 이모지] [동사로 시작하는 구체적 제목]`** — 우선순위 이모지를 **맨 앞**에.

| 이모지 | 라벨 | 의미 |
|:---:|---|---|
| 🔴 | P0 | 마일스톤/데모를 막음 — 최우선 |
| 🟠 | P1 | 높음 |
| 🟡 | P2 | 중간 |
| 🟢 | P3 | 낮음 |

- ✅ `🟡 Path Traversal 랩을 추가한다` · ❌ `XSS`, `버그 수정`
- (선택) 분류 태그 병기: `[구현] [결정] [교차검증] [자료조사] [현황]`.

## B3. 라벨 규칙 — P·type·area 각각 최소 1개
- **우선순위**: `P0`/`P1`/`P2`/`P3` (제목 이모지와 일치)
- **type**: `feature`/`bug`/`docs`/`ui`/`chore`/`idea`
- **area**: `backend`/`frontend`/`lens`/`visualization`/`infra`/`docs`
- **상태**(필요 시): `needs-decision`/`needs-spike`/`blocked`

## B4. 본문 템플릿 — 세 절 고정
```markdown
## 문제
- 무엇이 왜 문제인가. 배경·제약·불변식을 구체적으로. 관련 파일 경로·타입·기존 동작을 적는다.

## 제안
- 어떻게 할 것인가. 결정 이슈면 옵션 A/B + 권장안. 결정 이슈는 "등록으로 코드가 바뀌지 않음. 팀 결정만 요청" 명시.

## 검증
- 완료를 어떻게 확인하나(테스트/수동 확인/불변식). 결정 이슈면: 결정이 근거와 함께 이 이슈에 기록됨 = 완료.

관련: #관련이슈, PR #번호, docs/파일
```
> **자기완결성**: 본문만으로 착수 가능해야 한다. 새 팀원이든 AI든 대화 없이 실행하게. (loupe #2가 좋은 예)
> GitHub 이슈 폼 템플릿(`.github/ISSUE_TEMPLATE/`)이 이 세 절을 자동으로 띄운다.

## B5. `needs-decision` 이슈 특칙
- 본문에 **"이 이슈는 아이디어를 기록하고 팀 결정을 요청. 등록으로 코드가 바뀌지 않음"** 명시.
- `## 검증` = **"결정(수락 + 목표일 / 명시적 보류 + 사유)이 이 이슈에 기록됨."** (예: loupe #61, #20)

## B6. 상태 관리
- 진행에 따라 라벨을 옮긴다(결정되면 `needs-decision` 제거 등).
- 코드로 닫히는 이슈는 PR 본문에 **`Closes #번호`**. 관련 이슈는 본문에서 `#번호`로 상호 링크.
- 상시 문서형 이슈(비전·온보딩·**시행착오 회고**)는 닫지 않고 열어둔다.

## B7. AI 에이전트에게 맡길 때 (프롬프트 관점)
- **범위를 하나로** 좁혀라. **완료 조건(`## 검증`)을 테스트 가능하게** 적어라.
- **파일 경로·타입·예시 입력/출력**을 본문에 넣어라.
- **유형·area에 맞는 스킬을 호출하라**(A6): 프론트·시각화면 `frontend-design`·`taste-skill`·`animate`·`visualize`, 검증이면 `code-review`, 자료조사면 `research`.
- 결정 지점은 옵션으로 제시하고, **AI가 임의로 결정하지 않도록** `needs-decision`을 붙여라.

## B8. 좋은 예 / 나쁜 예
- ✅ `🟠 Bind the Proven issuer to the verified signer certificate` (loupe #67) — 우선순위·동사·구체·검증 가능.
- ✅ `🟡 [교차검증] 공통 시각화 데이터 스키마 사전 검증` (우리 #5) — 검증 먼저, 스코프 명확.
- ❌ `XSS 고치기` — 우선순위·무엇·완료조건 전부 없음.

## 등록·착수·마무리 체크리스트

**기획안 → 등록**
- [ ] 스킬로 기획안을 완성했는가
- [ ] **사용자에게 "이슈로 등록할까요?" 물었는가** (임의 등록 금지)
- [ ] 등록 후 이후 계획을 제안했는가
- [ ] 이슈에 없는 내용이 나왔을 때 이슈화 여부를 물었는가

**이슈 작성**
- [ ] 제목이 **우선순위 이모지 + 동사 + 구체**인가
- [ ] P·type·area 라벨(결정/검증이면 needs-*)을 달았는가
- [ ] 본문이 `## 문제 / ## 제안 / ## 검증` 3절이며 대화 없이 착수 가능한가
- [ ] 관심사가 하나인가

**착수 → 마무리**
- [ ] 유형·area에 맞는 **스킬(A6)** 을 호출했는가
- [ ] **스파이크를 먼저** 걸어 리스크를 제거했는가 (실패 시 사용자에게 대안 제안)
- [ ] (팀) **내 이름 브랜치**에서 작업하고 **assignees**로 상태를 표시했는가
- [ ] 검증을 **기능별 이슈**로 나눴는가 (통합 검증은 맨 마지막)
- [ ] **시행착오를 이슈 댓글**로 남겼는가
- [ ] 완료 시 이슈를 닫았는가 (회고 이슈는 예외 — 절대 닫지 않음)

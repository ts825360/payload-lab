# 이슈 작성 가이드 (초안 v0)

> 이 저장소의 **모든 작업은 이슈에서 시작한다.** 기획·결정·검증·구현·버그·문서 — 무엇이든
> 먼저 이슈로 만든다. 이 문서는 "언제·어떻게 이슈를 쓰는가"의 규칙이며, **사람과 AI 에이전트
> 둘 다** 이 규칙대로 이슈를 만들고 실행한다.
>
> 참고: 우리 저장소(#1–#21) + [RomainHartmann/loupe](https://github.com/RomainHartmann/loupe)의
> 이슈 체계에서 더 나은 방식을 합쳐 만들었다.
>
> ⚠️ **초안입니다.** 검토 후 확정/영어 번역하세요.

---

## 0. 왜 이슈 중심으로 일하나

- 의사결정과 **그 근거**가 한곳에 남아 추적 가능하다.
- 새 팀원·AI가 대화 없이 **본문만 읽고 착수**할 수 있다.
- "종이 위에서 설계했다가 실제 데이터에 안 맞았던" 과거 실패를 막기 위해, **검증을 결정보다
  먼저** 이슈로 박는다. (이게 이 프로젝트의 핵심 규율 — 5절)

---

## 1. 언제 이슈를 만드나 — 하나의 이슈 = 하나의 관심사

| 상황 | type 라벨 | 제목 동사 예 |
|---|---|---|
| 새 기능/랩 구현 | `type: feature` | 구현한다 / Build / Implement |
| 계약·타입·인터페이스 정의 | `type: feature` | 정의한다 / Define |
| 착수 전 결정이 필요한 제안 | `type: idea` + `needs-decision` | 결정한다 / Decide |
| 결정 전 실제로 대봐야 하는 검증 | `type: idea`/`feature` + `needs-spike` | 검증한다 / 교차검증 / Verify |
| 버그 | `type: bug` | 고친다 / Fix |
| 문서 | `type: docs` | 문서화한다 / Document |
| 도구·설정·유지보수 | `type: chore` | 정리한다 / Set up |

원칙: **한 이슈는 한 가지 이유로만 바뀐다.** 관심사가 여럿 섞이면 나눈다.

---

## 2. 제목 규칙 — 이슈탭에서 한눈에 띄게

형식: **`[우선순위 이모지] [동사로 시작하는 구체적 제목]`**
우선순위 이모지를 **맨 앞**에 둬서, 제목만 봐도 이슈탭에서 우선순위가 바로 보이게 한다.

| 이모지 | 라벨 | 의미 |
|:---:|---|---|
| 🔴 | P0 | 마일스톤/데모를 막음 — 최우선 |
| 🟠 | P1 | 높음 |
| 🟡 | P2 | 중간 |
| 🟢 | P3 | 낮음 |

- 반드시 **동사로 시작**하고 **구체적**으로.
  - ✅ `🟡 Path Traversal 랩을 추가한다`, `🟢 배포 방식을 결정한다`
  - ❌ `XSS`, `버그 수정`, `개선` — 무엇을·왜가 없다.
- (선택) 앞에 분류 태그를 붙여도 좋다: `[구현] [결정] [교차검증] [자료조사] [현황]`.

---

## 3. 라벨 규칙 — P·type·area 각각 최소 1개

- **우선순위**: `P0` / `P1` / `P2` / `P3` (제목 이모지와 일치시킬 것)
- **type**: `feature` / `bug` / `docs` / `ui` / `chore` / `idea`
- **area**: `backend` / `frontend` / `lens` / `visualization` / `infra` / `docs` (해당하는 것 모두)
- **상태**(필요 시): `needs-decision` / `needs-spike` / `blocked`

---

## 4. 본문 템플릿 — 세 절 고정

```markdown
## 문제
- 무엇이 왜 문제인가. 배경·제약·불변식(invariant)을 구체적으로.
- 관련 파일 경로·타입·기존 동작을 적어, 읽는 사람이 맥락을 다시 물어보지 않게 한다.

## 제안
- 어떻게 할 것인가. 결정 이슈면 옵션 A / B + 권장안을 제시.
- 결정 이슈는 "등록 자체로 코드가 바뀌지 않음. 팀 결정만 요청"을 명시한다.

## 검증
- 완료를 어떻게 확인하나 (테스트 / 수동 확인 / 불변식).
- 결정 이슈면: 결정이 이 이슈에 근거와 함께 기록됨 = 완료.

관련: #관련이슈, PR #번호, docs/파일경로
```

> **핵심 원칙 — 자기완결성(self-contained):** 본문만으로 착수 가능해야 한다. 파일 경로·타입·
> 예시 입력을 적어라. 새 팀원이든 AI 에이전트든 **대화 없이 실행**할 수 있게. (loupe #2가 좋은 예)

---

## 5. 핵심 규율 — 검증 후 결정 (verify-before-decide)

- 실제 데이터/입력에 대봐야 알 수 있는 것은, **`needs-spike` 검증 이슈를 먼저** 만들고
  그 결과로 **`needs-decision` 결정 이슈를 닫는다.**
- 규칙·스키마·인터페이스는 **현실적·적대적 예시 입력에 대보기 전에는 신뢰하지 않는다.**
- 예: 우리 #4(Command Injection 안전 설계), #5(시각화 스키마), #9(Lens 규칙 체인),
  loupe #22(end-to-end 스파이크), #54(라이브 API 검증).

---

## 6. `needs-decision` 이슈 특칙

- 본문에 **"이 이슈는 아이디어를 기록하고 팀 결정을 요청한다. 등록으로 코드가 바뀌지 않는다"** 명시.
- `## 검증` = **"결정(수락 + 목표일 / 명시적 보류 + 사유)이 이 이슈에 기록됨."**
- 예: loupe #61(스페인어 지원 여부), #20(배포 방식 결정).

---

## 7. 상태 관리

- 진행에 따라 라벨을 옮긴다(예: 결정되면 `needs-decision` 제거).
- 코드로 닫히는 이슈는 **PR 본문에 `Closes #번호`** 로 연결 → 머지 시 자동 종료.
- 관련 이슈는 본문에서 `#번호`로 상호 링크.
- 상시 문서형 이슈(비전·온보딩)는 닫지 않고 열어둔다.

---

## 8. AI 에이전트에게 맡길 때 (프롬프트 관점)

이슈를 AI가 실행하게 하려면 **본문이 곧 프롬프트**다:

- **범위를 하나로** 좁혀라 (한 이슈 = 한 관심사).
- **완료 조건(`## 검증`)을 테스트 가능하게** 적어라 — "된다"의 정의가 애매하면 AI도 사람도 못 끝낸다.
- **파일 경로·타입·예시 입력/출력**을 본문에 넣어라.
- 결정이 필요한 지점은 옵션으로 제시하고, **AI가 임의로 결정하지 않도록** `needs-decision`을 붙여라.

- ✅ `🟠 Bind the Proven issuer to the verified signer certificate` (loupe #67) — 우선순위·동사·구체·검증 가능.
- ✅ `🟡 [교차검증] 공통 시각화 데이터 스키마 사전 검증` (우리 #5) — 검증 먼저, 스코프 명확.

---

## 10. 새 이슈 체크리스트 (등록 전 확인)

- [ ] 제목이 **우선순위 이모지 + 동사 + 구체**인가
- [ ] P·type·area 라벨을 붙였는가 (결정/검증이면 needs-* 도)
- [ ] 본문이 `## 문제 / ## 제안 / ## 검증` 3절인가
- [ ] 본문만으로 **대화 없이 착수 가능**한가 (파일·타입·예시 포함)
- [ ] 관심사가 하나인가 (여럿이면 나눴는가)
- [ ] 검증이 필요한 결정이면 spike를 **먼저** 걸었는가

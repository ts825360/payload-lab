# PayloadLab 개발 여정과 시행착오

> 이 문서는 기획부터 현재까지의 **전체 과정과 시행착오**를 남긴 회고 기록이다.
> "왜 이렇게 만들었나"와 "무엇을 틀렸다가 고쳤나"를 함께 남겨, 새 팀원과 심사에서
> 결과물만이 아니라 **과정**을 볼 수 있게 한다. (관련: 이슈 #1–#20)

## 0. 한 줄 정의

DVWA 같은 웹 공격 실습 도구를 참고하되, **왜 공격이 성공했는지 / 왜 실패했는지**를
초보자도 이해하게 만드는 교육용 플랫폼. 성공하면 입력이 서버에서 처리되는 과정을
다이어그램으로, 실패하면 어떤 조건이 빠졌는지 **Lens**가 짚어준다.

- 스택: Python(FastAPI) · React(Vite) · SQLite · Docker
- 현재: SQL Injection · Reflected XSS · IDOR · Command Injection, 각 Easy/Medium = **8개 랩** 동작

---

## 1. 방법론 — "검증 후 결정"이 이 프로젝트의 뼈대

이전 캡스톤에서 팀이 **탐지 규칙을 실제 페이로드 구조에 대보지 않고 종이 위에서만
설계**했다가 규칙이 하나도 안 맞았던 실패를 겪었다. 그 반작용으로 이 프로젝트는
모든 설계 이슈를 **기획 → 자료조사 → 검증(spike) → 결정** 순서로, **검증을 결정보다
먼저** 배치했다. `needs-spike` 이슈를 관련 `needs-decision` 이슈보다 앞에 두는 것이
그 규율의 구현이다.

이 원칙은 이후 모든 단계에서 반복해서 값을 했다(아래 시행착오 참고).

## 2. 단계별 기록

### 2-1. 기획 (이슈 #1, #2, #12, #13)
- 프로젝트 이름: `attack-lens` → **PayloadLab**로 변경(#1). 기존 URL은 리다이렉트.
- MVP 범위 확정(#2): SQLi / Reflected XSS / IDOR, 각 Easy/Medium.
  - **Medium = Easy와 같은 핵심 기법 + 순진한 필터 하나를 우회.** 다른 기법이 아니라
    "필터 하나 더"로 좁게 잡아, 각 랩이 규칙 체인 하나 + 규칙 한 개 확장으로 끝나게 함.
- Command Injection·Stored XSS·DOM XSS·CSRF는 Phase 2로 유보.
- 비전/온보딩 문서(#12, #13)를 코드보다 먼저 작성.

### 2-2. 검증 스파이크 (이슈 #3–#9) — 여기서 시행착오가 가장 많았다
결정 전에 인터페이스를 실제/적대적 입력에 대본 결과:
- **#4 Command Injection 안전 설계**: 실제 subprocess 없이 **순수 시뮬레이션(Option A)**
  으로 충분함을 확인. 단, 진짜 bash는 `;;`/`|||` 등에 더 엄격 → "알려진 단순화"로 문서화.
- **#5 시각화 스키마**: 공통 6단계 스키마가 SQLi/XSS엔 잘 맞지만 **IDOR("빠진 검사"류)
  에선 `transformation` 필드 의미가 늘어남**을 발견. → 나중에 실행 그래프 재설계의 씨앗.
- **#9 Lens 규칙 체인**: `RuleCheck`/`evaluate_chain` 인터페이스를 SQLi·IDOR에 검증.
  **규칙 체인은 자신이 설계된 그 기법 하나만 인식**함을 확인(SQLi boolean-bypass 체인이
  stacked-query DROP TABLE을 오진). → 좁은 MVP 스코프와 일관된 한계로 수용.
- **#3 모듈 경계**: #5+#9를 하나의 `Lab`/`AttemptResult` 인터페이스로 통합. `lens_passed_count`
  가 #8(부분 성공 체크리스트)이 필요로 한 데이터를 **공짜로** 제공함을 발견.

### 2-3. 결정 (이슈 #1, #2, #6, #7, #8, #11)
- #6 Lens 폴백: "규칙 미매칭"이 아니라 **`rule.check()`가 예외를 던지는 경우**로 재정의
  (체인은 구조적으로 항상 무언가를 반환하므로).
- #11 DVWA 차별점: **"소스-실행 연결"** 로 재정의 — 취약 타깃이 아니라 *왜 통했나*를
  보여주는 것이 차별점.

### 2-4. 구현 (MVP)
- FastAPI 백엔드 + React 프론트 스캐폴딩 → DVWA식 다크/라이트 레이아웃.
- **실제 오라클**: SQLi는 진짜 :memory: SQLite에 진짜 취약 쿼리 실행(패턴매칭 아님),
  XSS는 서버가 실제 HTML을 돌려주고 프론트가 iframe으로 **진짜 내비게이션**해 `<script>`가
  실제 실행됨(DVWA와 동일 원리), IDOR는 세션 대신 클라이언트 값을 신뢰하는 취약점.
- 세 랩에 Medium 난이도 추가(#ff55134).
- **#10 Lens 회귀 테스트 스위트**를 만들다가 `resolve()`가 악성 입력에 크래시하던
  **실제 버그를 발견·수정**. 테스트가 값을 한 첫 사례.

### 2-5. 인터페이스 대개조 (이번 세션 전반부)
> 이전 작업은 "인터페이스 수정이 어려워 핵심 로직만 여러 번 고쳤다"는 한계가 있었음.
> 이번에 UI/UX를 정면으로 다룸.
- Space Grotesk + IBM Plex 타이포, **보라(액션)/앰버(Lens)/민트(성공)** 토큰 체계,
  애니메이션 해·달 토글, 모든 다이어그램 재설계(다크·라이트).
- **리뷰 수정 4건**: 빈 입력·키보드로도 Lens 도달(#1), XSS 시각화가 실제 `GET /render`
  표기(#3), 로컬 전용 근거 문서화(#4), IDOR 죽은 규칙 제거(#18).

### 2-6. 실행 다이어그램 재설계 (이슈 #19 / PR #20) — 핵심 차별점 강화
"서버상태/코드/처리흐름 3탭"이 하나의 실행을 셋으로 쪼개 학습 흐름을 끊는다는 문제를,
**한 번의 스크롤로 이어지는 관계 그래프**로 통합. 매 시도의 **실제 데이터**로 채우되
오라클은 그대로(그래프는 그 위의 서술).
- SQLi: 쿼리가 단계별로 변형되는 유도 과정 + 실제 반환행
- XSS: 서버→브라우저 경계선 + 마지막에 실제 실행 iframe
- IDOR: 관계 스냅샷(점선 유령 노드 + owner 화살표, 내 것=초록/남의 것=빨강)
- Command Injection: 파이프라인형 — **렌더 코드 0줄 변경**으로 확장(스키마 확장성 실증)
- Payload ↔ 코드 ↔ 다이어그램 **양방향 하이라이트**(공유 group id 한 벌)

## 3. 시행착오 모음 (핵심만)

과정에서 실제로 틀렸다가 고친 것들 — 심사와 회고의 핵심.

| # | 처음 시도 | 문제 | 최종 |
|---|-----------|------|------|
| 이름 | attack-lens | 범위·톤이 안 맞음 | **PayloadLab** (#1) |
| Lens 노출 | 버튼 클릭 | 흐름 끊김 | 밑줄 hover→팝오버, 버튼을 팝업에 접음 (git: acc8c6c) |
| 밑줄 렌더 | 입력창 전체 폭 | 부정확·과함 | **실제 텍스트 길이**만큼, 빈칸엔 **최소 폭**(#1 리뷰) |
| 시각화 | 가짜 단계 파이프라인 | 결과만 보여줌 | 실제 state/code/live 뷰 → 최종 **유도 과정 그래프** |
| 설명 방식 | "문장 비유"로 쉽게 | 도구가 싸구려로 보임·"말하기"지 "보여주기" 아님 | **실제 쿼리가 변형되는 걸 보여주기**로 폐기·전환 |
| 스키마 | `nodes/edges/links[]` | 손 배선 복잡 | **steps 유니온 + 공유 group**으로 단순화(효과 동일) |
| Command Injection | (검증) `& `·개행 미인식 | **작동하는 공격을 "실패"로 오진** | 구분자에 `&`·개행 추가 |
| Command Injection | "실제 출력/실행됨" 표기 | 유일한 시뮬레이션 랩인데 진짜인 척 | **"예상 출력(시뮬레이션)"** 으로 정직화 |
| XSS Lens | `<script>`만 성공 판정 | `<img onerror>`(작동하는 XSS)를 "틀렸다"고 | 스코프 유지하되 **Lens 문안 정직화** |
| IDOR medium | claimed_user_id를 note로만 | 스펙은 화살표 원함 | **실제 관계 화살표**로 추가 |

또한 이 프로젝트의 **검증 규율 자체가 시행착오를 앞단에서 잡았다**: #5에서 IDOR이
공통 스키마를 늘린다는 걸, #9에서 규칙 체인이 한 기법만 인식한다는 걸, 실행 그래프
스파이크에서 naive 파이프라인이 XSS·IDOR에서 깨진다는 걸 **만들기 전에** 발견.

## 4. 검증·품질 게이트

- **테스트**: 백엔드 39개(그래프 생성·상황 반영·안전·규칙 진단), 프론트 9개(양방향
  하이라이트·live iframe·관계 렌더 회귀). Vitest는 호스트에 node가 없어 Node 컨테이너로 실행.
- **Lens 감사**: "작동하는 공격을 틀렸다고 하는" 케이스를 전 랩에 대해 적대적 입력으로
  점검 → CmdInjection `&`/개행, XSS `<img onerror>` 정직화로 해결. SQLi/IDOR은 실제
  오라클이라 거짓 오진 없음.
- **코드 리뷰(2축)**: 표준(Fowler smell)·스펙(#19) 병렬 감사 → 필터 단계 중복 추출,
  스키마 문서 동기화, IDOR claimed 화살표 등 팔로우업 반영.
- **안전성**: Command Injection은 `subprocess`/`os.system`/`eval` 호출 0건(문자열
  시뮬레이션만). `/render`는 실제 reflected XSS라 **로컬 전용** 근거를 README에 명시.

## 5. 정직한 한계 (심사에서 숨기지 않음)

- **좁은 스코프**: 각 랩은 한 기법 + 필터 하나. 규칙 체인은 그 기법만 인식(#9).
- **Command Injection은 시뮬레이션**: 안전을 위해 실제 셸 미실행. UI에 명시.
- **IDOR은 데모용 고정 세션**: 실제 인증/세션이 아니라 고정 사용자·주문 시드.
- **XSS 오라클은 `<script>` 한정**: 다른 벡터도 render는 실행하지만 판정은 안 함(문안으로 안내).
- **문안 미확정**: Lens/README 교육 문안은 자료조사 후 다듬을 예정(#14–#16).
- **단독 진행**: 팀원 합류 전이라 현재 1인 진행(온보딩 문서 #13 준비됨).
